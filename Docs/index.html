Deploy Manager

Complete setup and administration guide — from a fresh install to imaging your first machine.

1What is Deploy Manager

Deploy Manager is a self-hosted Windows deployment server. It PXE-boots bare-metal machines into WinPE, applies a Windows image (WIM), joins the machine to Active Directory in the correct OU based on its network subnet, installs software packages, and reports progress live to a web dashboard.

  • Zero-touch imaging — technicians create a deployment job in the web UI; the machine does the rest.
  • Site-aware — IP subnet determines the AD OU, timezone, and locale for each machine.
  • Software deployment — post-install packages run automatically after Windows setup.
  • Live monitoring — job status streams to the dashboard in real time.
  • Self-contained — a single MSI installs everything: web app, TFTP server, tray monitor. No external database, no IIS, no .NET prerequisite.

2How it works

The PXE boot chain, end to end:

Client PXE bootUEFI / BIOS firmware
→
DHCPoption 66 + 67
→
tftpd64serves the boot file (UDP 69)
→
shimMicrosoft-signed (Secure Boot)
→
iPXEfetches boot.ipxe over HTTP
→
wimbootloads WinPE at LAN speed (HTTP 8080)
→
WinPEruns Start-OSDeploy.ps1

On UEFI clients the firmware first loads ipxe-shim.efi — a Microsoft-signed shim — which in turn loads the iPXE-signed ipxe.efi. This satisfies UEFI Secure Boot with no firmware changes or key enrollment (see section 9). Legacy BIOS clients load undionly.kpxe directly.

Inside WinPE, Start-OSDeploy.ps1 downloads live configuration from the server (/api/deploy-config), reports the machine to the server (it appears under Machines), then polls http://server:8080/jobs/<mac>.json for a deployment job. When a job exists it downloads and applies the WIM, injects drivers, pre-stages the computer account in AD, and reboots into Windows. A post-install phase installs the software package and reports success or failure back to the API.

3Requirements

  • Windows Server 2019/2022 (or Windows 10/11) with a static IP address
  • Local administrator rights for installation
  • Access to configure your DHCP server (options 66 and 67 on the target scope)
  • Disk space for your images — a typical Windows 11 WIM is 4–6 GB; allow 20 GB+ on the drive holding C:\ProgramData\DeployManager\images
  • Windows ADK + WinPE add-on on the server (needed to build and patch the boot image — section 7). The signed iPXE Secure Boot chain and wimboot are already bundled in the installer, so no separate iPXE download is required.
  • An AD service account with Full Control over Computer objects in your target OUs (for domain join and re-imaging — see section 5)
  • No .NET installation is required — the runtime ships inside the application.

Network ports (opened automatically by the installer)

1

2
Finish On the final page, leave Launch the Deploy Manager tray monitor ticked. The installer starts both Windows services automatically.
3
Verify 
sc.exe query DeployManager
sc.exe query Tftpd64
Both should report RUNNING. The tray icon should be visible near the clock.

msiexec /i DeployManager-<version>-Setup.msi /qn

6Core settings & sites

Sites (Settings → Sites)

A site maps an IP subnet to an Active Directory OU, with an optional timezone and locale override. When a machine PXE boots, its IP determines its site, and therefore where the computer object is created and which regional settings are applied.

FieldExample
NameHead Office
Subnet192.168.10.0/24
OUOU=Workstations,OU=HeadOffice,DC=corp,DC=contoso,DC=com
Timezone / LocaleLeave blank to inherit the defaults

Configure at least one site before your first deployment.

Service account (Settings)

Enter the UPN and password of the AD account used to pre-stage computer objects (djoin) during imaging and to remotely trigger reboots via WMI. The password is encrypted at rest (AES-256-GCM).

The account does not need Domain Admin. Delegate the following on each target OU (e.g. via Active Directory Users and Computers → Delegate Control or dsacls):

PermissionApplies toPurpose
Create/Delete Computer ObjectsThis object and all descendantsCreate new machine accounts and clean up old ones
Full ControlDescendant Computer objectsRead/write all properties, reset passwords, take ownership (required by KB5020276 account-reuse hardening for re-imaging)

Additionally, the account needs local administrator rights on target machines for the PXE Reboot and Pre-stage Reboot actions (SMB to admin$ + WMI process creation).

WinPE local account (Settings)

Set the local administrator account name and password that WinPE bakes into deployed machines via unattend.xml. The password is never served over the network — it is embedded into boot.wim when you patch it (section 11), so re-patch whenever you change it.

7Install the Windows ADK

The ADK provides the tooling to build and service WinPE images. Two installers are required, in this order, on the Deploy Manager server:

1
Windows ADK — during setup, only the Deployment Tools feature is required.
learn.microsoft.com/en-us/windows-hardware/get-started/adk-install
2
Windows PE add-on for the ADK — same download page, separate installer. Provides the base WinPE image and copype.
Always service WinPE images with ADK DISM (C:\Program Files (x86)\Windows Kits\10\Assessment and Deployment Kit\Deployment Tools\amd64\DISM\dism.exe), not the inbox C:\Windows\System32\dism.exe — the inbox DISM cannot service current WinPE images and fails with obscure errors. All commands below assume the Deployment and Imaging Tools Environment prompt (Start Menu → Windows Kits), which puts ADK DISM first on the PATH.

8Build the WinPE boot image

Run everything below in an elevated Deployment and Imaging Tools Environment prompt.

1
Create a working copy of WinPE 
copype amd64 C:\WinPE
2
Mount the boot image 
dism /Mount-Image /ImageFile:C:\WinPE\media\sources\boot.wim /Index:1 /MountDir:C:\WinPE_MOUNT
(Create C:\WinPE_MOUNT first if it does not exist.)
3
Add PowerShell support — the task sequence is PowerShell, so these optional components are required (order matters): 
set PKG=C:\Program Files (x86)\Windows Kits\10\Assessment and Deployment Kit\Windows Preinstallation Environment\amd64\WinPE_OCs
dism /Image:C:\WinPE_MOUNT /Add-Package /PackagePath:"%PKG%\WinPE-WMI.cab"
dism /Image:C:\WinPE_MOUNT /Add-Package /PackagePath:"%PKG%\WinPE-NetFX.cab"
dism /Image:C:\WinPE_MOUNT /Add-Package /PackagePath:"%PKG%\WinPE-Scripting.cab"
dism /Image:C:\WinPE_MOUNT /Add-Package /PackagePath:"%PKG%\WinPE-PowerShell.cab"
dism /Image:C:\WinPE_MOUNT /Add-Package /PackagePath:"%PKG%\WinPE-StorageWMI.cab"
dism /Image:C:\WinPE_MOUNT /Add-Package /PackagePath:"%PKG%\WinPE-DismCmdlets.cab"
4
Add network drivers for your hardware — WinPE must get a DHCP lease on every model you deploy. Download the vendor's WinPE driver pack (Dell WinPE CAB, HP WinPE pack, Lenovo SCCM pack), extract it, then: 
dism /Image:C:\WinPE_MOUNT /Add-Driver /Driver:C:\WinPE_Drivers /Recurse
Do not use /ForceUnsigned. If a driver will not load without it, get the properly signed WinPE driver from the vendor instead — unsigned drivers fail on Secure Boot hardware anyway.
5
Copy in the Deploy Manager task sequence 
mkdir C:\WinPE_MOUNT\Deploy
copy "C:\Program Files\DeployManager\scripts\Start-OSDeploy.ps1"    C:\WinPE_MOUNT\Deploy\
copy "C:\Program Files\DeployManager\scripts\Start-PostInstall.ps1" C:\WinPE_MOUNT\Deploy\
6
Make WinPE launch the task sequence at boot — replace C:\WinPE_MOUNT\Windows\System32\startnet.cmd with: 
wpeinit
powershell -ExecutionPolicy Bypass -NoProfile -File X:\Deploy\Start-OSDeploy.ps1
7
Commit and copy to the TFTP root 
dism /Unmount-Image /MountDir:C:\WinPE_MOUNT /Commit

mkdir "%ProgramData%\DeployManager\tftp\Boot"
copy C:\WinPE\media\sources\boot.wim "%ProgramData%\DeployManager\tftp\Boot\boot.wim"
copy C:\WinPE\media\Boot\BCD         "%ProgramData%\DeployManager\tftp\Boot\BCD"
copy C:\WinPE\media\Boot\boot.sdi    "%ProgramData%\DeployManager\tftp\Boot\boot.sdi"
The TFTP root now contains Boot\boot.wim, Boot\BCD and Boot\boot.sdi. The Settings → Boot Image page should show "boot.wim found".

9iPXE binaries & TFTP root

Deploy Manager uses iPXE to move the boot process off slow TFTP and onto HTTP as early as possible.

Nothing to do here — the installer already staged these files. The signed iPXE Secure Boot chain and wimboot ship inside the MSI and are placed in the TFTP root during installation. This section is reference; skip to section 10 (DHCP) unless you want to update the binaries.

To pull a newer signed iPXE release later (optional), run in an elevated PowerShell on the server:

powershell -ExecutionPolicy Bypass -File "C:\Program Files\DeployManager\scripts\Get-IpxeBinaries.ps1"

It downloads the latest signed chain and overwrites the bundled copies in place. The version that shipped is recorded in C:\ProgramData\DeployManager\tftp\VERSION.txt.

The files staged in the TFTP root:

FilePurpose
ipxe-shim.efiUEFI boot file (DHCP option 67). A Microsoft-signed shim; loads ipxe.efi. Works with Secure Boot on or off.
ipxe.efiiPXE with the full driver set, signed by the iPXE Secure Boot CA. Loaded by the shim — not booted directly.
snponly-shim.efi / snponly.efiAlternative chain using iPXE's UEFI SNP-only driver. Try this if a NIC misbehaves with the full ipxe.efi.
undionly.kpxeiPXE for legacy BIOS clients (no Secure Boot involved)
wimbootSecure Boot-capable WIM loader — loads WinPE over HTTP

When an iPXE client asks the server for http://<server>:8080/boot.ipxe, Deploy Manager generates the boot script on the fly from current settings — there is no boot script file to maintain.

Final TFTP root layout (C:\ProgramData\DeployManager\tftp):

tftp\
├── ipxe-shim.efi      # UEFI boot file (Microsoft-signed shim)
├── ipxe.efi           # loaded by the shim (iPXE-signed)
├── snponly-shim.efi   # alt chain
├── snponly.efi
├── undionly.kpxe      # legacy BIOS
├── wimboot
└── Boot\
    ├── BCD
    ├── boot.sdi
    └── boot.wim
UEFI Secure Boot works out of the box — no firmware changes, no key enrollment, no disabling Secure Boot. Since iPXE v2.0.0 the project ships a Microsoft-signed shim (ipxe-shim.efi, signed by the "Microsoft Windows UEFI Driver Publisher" CA that firmware already trusts). The shim loads ipxe.efi, which is signed by the iPXE Secure Boot CA embedded in that shim. The trust chain is:

Firmwaretrusts Microsoft UEFI CA
→
ipxe-shim.efiMicrosoft-signed
→
ipxe.efiiPXE-CA signed
→
wimboot → WinPEMicrosoft-signed boot mgr

Point DHCP option 67 at ipxe-shim.efi for UEFI clients and you are done. The one trade-off: you must use the iPXE binaries exactly as shipped — you cannot compile a custom iPXE with extra features and keep it Secure Boot-trusted, because the shim only trusts binaries the iPXE project itself signs. The stock ipxe.efi covers everything Deploy Manager needs.

10DHCP configuration

Only two DHCP scope options are required. The installer pre-stages an autoexec.ipxe script in the TFTP root that automatically chains iPXE to the Deploy Manager HTTP boot script, so no DHCP user-class policies or vendor-class rules are needed.

1
Scope options — set these on the DHCP scope that covers the machines you want to PXE boot. In an elevated PowerShell on the DHCP server: 
# Replace 192.168.10.0 with YOUR scope and <server-ip> with the Deploy Manager server IP
Set-DhcpServerv4OptionValue -ScopeId 192.168.10.0 -OptionId 66 -Value "<server-ip>"
Set-DhcpServerv4OptionValue -ScopeId 192.168.10.0 -OptionId 67 -Value "ipxe-shim.efi"
OptionValuePurpose
066 — Boot Server Host NameThe Deploy Manager server's IP addressTells PXE where to TFTP from
067 — Bootfile Nameipxe-shim.efiMicrosoft-signed shim for Secure Boot
Use ipxe-shim.efi, not ipxe.efi, as the UEFI boot file. The shim is the Microsoft-signed first stage that Secure Boot firmware will accept; it then loads ipxe.efi itself. Pointing option 67 straight at ipxe.efi would be rejected on machines with Secure Boot enabled.
2
Verify — confirm the options are set: 
Get-DhcpServerv4OptionValue -ScopeId 192.168.10.0
You should see option 66 = your server IP and option 67 = ipxe-shim.efi.
3
Mixed UEFI/BIOS fleets (optional) — use DHCP policies on option 60/93 (client architecture) to serve ipxe-shim.efi to UEFI and undionly.kpxe to BIOS within the same scope.

How the boot chain works

No manual intervention needed — this is just background on what happens during PXE boot:

  1. Firmware PXE sends DHCP, gets option 66 (server IP) and option 67 (ipxe-shim.efi).
  2. Firmware downloads ipxe-shim.efi via TFTP — Microsoft-signed, passes Secure Boot.
  3. The shim loads ipxe.efi — signed by iPXE CA (trusted by the shim).
  4. iPXE fetches autoexec.ipxe from the TFTP server (pre-installed by the MSI).
  5. autoexec.ipxe chains to http://<server>:8080/boot.ipxe (using DHCP option 66 as the server address).
  6. The HTTP boot script downloads wimboot, BCD, boot.sdi, and boot.wim over HTTP at full LAN speed.
  7. WinPE loads and runs Start-OSDeploy.ps1.
Test: PXE-boot any machine on the scope. You should see iPXE load, fetch boot.ipxe, download boot.wim over HTTP at LAN speed, and land in WinPE. Within a minute the machine appears under Machines in the web UI.

11Patch boot.wim

Start-OSDeploy.ps1 inside the boot image ships with two placeholders: the server URL and the WinPE local password. They must be patched before the image is usable — otherwise WinPE has no idea where its server is.

1
In the web UI open Settings → Boot Image. The status strip must show boot.wim found, API URL configured, WinPE password configured and ADK DISM found.
2
Click Patch boot.wim. The server mounts the image with ADK DISM (mount point C:\WinPE_MOUNT), replaces the placeholders, and commits. Output appears on the page.
The patch also copies the latest Start-OSDeploy.ps1 and Start-PostInstall.ps1 into the WIM before replacing placeholders — so after an upgrade you only need to re-patch (no manual file copy).
Re-run the patch whenever the API server URL or the WinPE local password changes, or after upgrading Deploy Manager. The equivalent command line, if you prefer scripting: 
powershell -ExecutionPolicy Bypass -File "C:\Program Files\DeployManager\scripts\Update-BootWim.ps1" `
    -ServerUrl "https://<server-ip>:8090" -WinpePassword "<password>"

12Windows install images

Option A — straight from a Windows ISO

1
Mount the Windows ISO (double-click), note its drive letter (e.g. E:), then list the editions inside: 
dism /Get-WimInfo /WimFile:E:\sources\install.wim
2
Export the edition you deploy (e.g. index 6 = Pro, index 3 = Enterprise — check the listing) into the images directory: 
dism /Export-Image /SourceImageFile:E:\sources\install.wim /SourceIndex:6 `
     /DestinationImageFile:"%ProgramData%\DeployManager\images\Win11-24H2-Pro.wim" /Compress:max
If the ISO contains install.esd instead of install.wim, the same command works — DISM converts ESD to WIM during export.

Option B — a customised, captured image

Build a reference machine (install Windows, apps, updates), generalise it with sysprep /oobe /generalize /shutdown, boot it into WinPE, and capture:

dism /Capture-Image /ImageFile:D:\GoldenImage.wim /CaptureDir:C:\ /Name:"Golden Win11" /Compress:max

Copy the WIM into %ProgramData%\DeployManager\images\.

Register the image

Web UI → Images → Add WIM Image: give it a display name and the path relative to the images directory (e.g. Win11-24H2-Pro.wim). Only Enabled images are offered when creating jobs.

13Driver packages

Driver packages are injected into the applied Windows image during deployment, so each model gets its correct drivers without touching the golden image.

1
Download the vendor's enterprise driver pack for the model (Dell Family CAB, HP driver pack, Lenovo SCCM pack) and extract it into a subfolder of C:\ProgramData\DeployManager\drivers, e.g. drivers\Latitude-5450\.
2
Web UI → Drivers → Add: name it and select the folder. When creating a deployment job you choose which driver package applies.

14Software & packages

Software entries describe a single installer; Packages group them into an ordered set applied after Windows setup.

1
Software → Add Software — for each application provide the installer location (a UNC path or a path under the images share), the working directory, and the silent install command, e.g. setup.exe /S or msiexec /i app.msi /qn.
2
Packages → Create Package — select the software items and their order, e.g. a "Standard SOE" package with your antivirus, office suite, VPN client and monitoring agent.

During deployment, each install's exit code is captured and reported to the job — failures are visible per-application in the Jobs page.

15Deploying a machine

1
PXE-boot the target machine once (or add it manually under Machines → Add Machine with its MAC address). Discovered machines report their MAC, IP, model and serial number automatically.
2
Create the job — Deploy → New Deployment: pick the machine, confirm the device name (prefix is suggested), site and OU (auto-detected from subnet), choose the WIM image, software package, driver package, and join mode (Domain Join for on-prem AD, or Workgroup for Autopilot/Intune — the machine boots to OOBE for Entra enrollment). Then choose one of three actions:
  • Save Job Only — saves the job file. You manually PXE-boot the machine (e.g. via BIOS boot menu).
  • Save Job + PXE Reboot — saves the job, then remotely sets the machine's next boot to its PXE/NIC firmware entry and reboots it via WMI. Requires the service account to have local admin rights on the target.
  • Save Job + Pre-stage Reboot — saves the job, then remotely stages WinPE to the machine's local disk and reboots into it via the Windows Boot Manager. No PXE required — works over the WAN and on models where PXE is unavailable or blocked by Secure Boot policy. Also requires local admin on the target.
3
WinPE loads and images the machine. It finds its pending job within seconds and starts imaging — partitioning, applying the WIM, injecting drivers, joining the domain (or workgroup for Autopilot), writing unattend.xml with the site's timezone/locale, then rebooting into Windows setup and the software phase. Workgroup deployments boot into OOBE for Autopilot enrollment instead of the desktop.
4
Watch it live — the Jobs page streams status in real time: pending → imaging → software installs (per-app progress) → complete. Stale jobs are timed out automatically by the watchdog. Email notifications on completion/failure can be enabled in Settings.
That's the full loop. From here, imaging a machine is: create job → PXE boot → wait for the green badge.

16HTTPS certificate

A 5-year self-signed certificate is generated on first start. To replace it with one signed by your internal CA (recommended — removes browser warnings and enables full certificate validation from WinPE):

1
Settings → Certificate → Generate CSR — enter the server's DNS name(s) and IP(s) as subject alternative names. The private key never leaves the server (stored encrypted until the signed certificate is installed).
2
Sign the CSR with your CA — for AD CS: 
certreq -submit -attrib "CertificateTemplate:WebServer" deploymanager.csr
3
Upload the signed certificate (PEM/Base64) plus any chain on the same page. It is served to new connections immediately — no service restart needed. The dashboard warns 30 days before expiry.

17Users & authentication

  • Breakglass login — the password set during first-run setup, always available at /auth/login. Login attempts are rate-limited (10 per 15 minutes per IP).
  • Microsoft Entra ID SSO — configure Tenant ID, Client ID and Client Secret in Settings (create an app registration with redirect URI https://<server>:8090/signin-oidc). If you access the server via multiple hostnames or IPs, add a redirect URI for each (e.g. https://deploymanager.example.com:8090/signin-oidc and https://192.168.1.10:8090/signin-oidc). Only users you have added under Users (by UPN) can sign in, each with a role controlling their access.
    Restart the Deploy Manager service after changing the sign-in method, Tenant ID, Client ID or secret. These settings are read once at startup and do not take effect until the service is restarted (via the tray app or sc.exe restart DeployManager).
  • Audit log — every sign-in, settings change, certificate operation and deployment action is recorded, viewable under Audit.

18Operations & backup

Backup

Everything that matters lives in C:\ProgramData\DeployManager — back up this folder. Critical files within it:

PathContents
data\deploymgr.dbSQLite database — machines, jobs, images, packages, users
data\settings.jsonConfiguration (secrets encrypted)
data\enc.keyThe AES-256 encryption key. Without it, stored secrets are unrecoverable — protect this backup accordingly
data\deploymgr.pfxThe HTTPS certificate and private key
images\ / drivers\ / tftp\Your WIMs, driver packs and boot files

Health monitoring

https://<server>:8090/health returns unauthenticated JSON with service status, certificate expiry and disk space — point your monitoring platform at it.

Upgrades

Run the newer MSI over the top — it upgrades in place. Services are stopped and restarted; your data in ProgramData is untouched.

Uninstall

Removes the program, services and firewall rules. C:\ProgramData\DeployManager is deliberately left behind (your database, images and keys) — delete it manually if you want a complete removal.

19Troubleshooting

SymptomCheck
Web UI unreachablesc.exe query DeployManager. Service logs are in Event Viewer → Application, sources DeployManager and .NET Runtime. Remember: the UI is HTTPS on 8090 — port 8080 serves files only and returns 404 for everything else.
PXE client says "no boot filename received"DHCP options 66/67 not set on the scope, or a firewall between client and DHCP. Verify with ipconfig /all on a machine in the scope.
iPXE loads, then loops back into iPXEVerify autoexec.ipxe exists in the TFTP root (%ProgramData%\DeployManager\tftp\). The installer places it there; if deleted, recreate it with: #!ipxe newline chain http://${next-server}:8080/boot.ipxe.
TFTP timeout downloading the boot filesc.exe query Tftpd64; confirm UDP 69 open (installer adds the rule); confirm ipxe-shim.efi exists in the TFTP root. Test from another machine: tftp -i <server> get ipxe-shim.efi. If the service runs but serves nothing, check that the config file is named tftpd32.ini (legacy filename) — the service edition ignores tftpd64.ini.
Secure Boot violation loading the boot fileConfirm DHCP option 67 is ipxe-shim.efi (the Microsoft-signed shim), not ipxe.efi directly. Re-run Get-IpxeBinaries.ps1 if the shim is missing from the TFTP root. If a specific NIC still fails, try the snponly-shim.efi chain.
iPXE says "BCD not found" or "boot.sdi not found"The TFTP Boot folder is missing BCD or boot.sdi. Copy them from your WinPE build directory: copy C:\WinPE\media\Boot\BCD "%ProgramData%\DeployManager\tftp\Boot\" and the same for boot.sdi. See section 8, step 7.
WinPE boots but "CANNOT REACH DEPLOYMANAGER SERVER"The boot.wim was not patched (section 11), the API URL in it is stale, or WinPE has no NIC driver (section 8, step 4) — check ipconfig in the WinPE console.
Machine images but does not join the domainService account credentials in Settings, its Full Control delegation on Computer objects in the target OU (section 5), and that the site's OU DN is valid. For re-images, the service account must also be the owner of the existing computer object (KB5020276).
Software installs report failuresExpand the job's software results in the Jobs page — each row has the installer's exit code and error output.
Service fails to start after config changesAlmost always an invalid appsettings.json or an in-use port. The exact exception is in Event Viewer → Application, source .NET Runtime.
