Deploy Manager

Complete setup and administration guide — from a fresh install to imaging your first machine.

1What is Deploy Manager

Deploy Manager is a self-hosted Windows deployment server. It PXE-boots bare-metal machines into WinPE, applies a Windows image (WIM), joins the machine to Active Directory in the correct OU based on its network subnet, installs software packages, and reports progress live to a web dashboard.

  • Zero-touch imaging — technicians create a deployment job in the web UI; the machine does the rest.
  • Site-aware — IP subnet determines the AD OU, timezone, and locale for each machine.
  • Software deployment — post-install packages run automatically after Windows setup.
  • Live monitoring — job status streams to the dashboard in real time.
  • Autopilot / Intune — deploy machines in workgroup mode for Autopilot enrollment, with optional automatic device registration via the Microsoft Graph API.
  • Self-contained — a single MSI installs everything: web app, TFTP server, tray monitor. No external database, no IIS, no .NET prerequisite.

2How it works

The PXE boot chain, end to end:

Client PXE bootUEFI / BIOS firmware
→
DHCPoption 66 + 67
→
tftpd64serves the boot file (UDP 69)
→
shimMicrosoft-signed (Secure Boot)
→
iPXEfetches boot.ipxe over HTTP
→
wimbootloads WinPE at LAN speed (HTTP 8080)
→
WinPEruns Start-OSDeploy.ps1

On UEFI clients the firmware first loads ipxe-shim.efi — a Microsoft-signed shim — which in turn loads the iPXE-signed ipxe.efi. This satisfies UEFI Secure Boot with no firmware changes or key enrollment (see section 9). Legacy BIOS clients load undionly.kpxe directly.

Inside WinPE, Start-OSDeploy.ps1 downloads live configuration from the server (/api/deploy-config), reports the machine to the server (it appears under Machines), then polls http://server:8080/jobs/<mac>.json for a deployment job. When a job exists it downloads and applies the WIM, injects drivers, pre-stages the computer account in AD, and reboots into Windows. A post-install phase installs the software package and reports success or failure back to the API; for Autopilot deployments it also collects the hardware hash, serial number and Windows Product ID for Intune registration.

3Requirements

  • Windows Server 2019/2022 (or Windows 10/11) with a static IP address
  • Local administrator rights for installation
  • Access to configure your DHCP server (options 66 and 67 on the target scope)
  • Disk space for your images — a typical Windows 11 WIM is 4–6 GB; allow 20 GB+ on the drive holding C:\ProgramData\DeployManager\images
  • Windows ADK + WinPE add-on on the server (needed to build and patch the boot image — section 7). The signed iPXE Secure Boot chain and wimboot are already bundled in the installer, so no separate iPXE download is required.
  • An AD service account with Full Control over Computer objects in your target OUs (for domain join and re-imaging — see section 6)
  • No .NET installation is required — the runtime ships inside the application.

6Core settings & sites

Sites (Settings → Sites)

A site maps an IP subnet to an Active Directory OU, with an optional timezone and locale override. When a machine PXE boots, its IP determines its site, and therefore where the computer object is created and which regional settings are applied.

FieldExample
NameHead Office
Subnet192.168.10.0/24
OUOU=Workstations,OU=HeadOffice,DC=corp,DC=contoso,DC=com
Timezone / LocaleLeave blank to inherit the defaults

Configure at least one site before your first deployment.

Service account (Settings)

Enter the UPN and password of the AD account used to pre-stage computer objects (djoin) during imaging and to remotely trigger reboots via WMI. The password is encrypted at rest (AES-256-GCM).

The account does not need Domain Admin. Delegate the following on each target OU (e.g. via Active Directory Users and Computers → Delegate Control or dsacls):

PermissionApplies toPurpose
Create/Delete Computer ObjectsThis object and all descendantsCreate new machine accounts and clean up old ones
Full ControlDescendant Computer objectsRead/write all properties, reset passwords, take ownership, and delete (needed to recreate an object when a re-image hits the KB5020276 re-use block)

Additionally, the account needs local administrator rights on target machines for the remote-reboot actions — see the next subsection.

Local admin for remote reboot (PXE Reboot / Pre-stage Reboot)

The Save Job + PXE Reboot and Save Job + Pre-stage Reboot actions restart a running, domain-joined machine into the deployment without anyone touching it. To do that, Deploy Manager (as the service account) authenticates to the target's admin$ administrative share to drop a small launcher, then uses WMI to run it. Administrative shares are only reachable by local administrators, so the service account must be a local admin on the target — otherwise the action fails with:

Could not authenticate to \\<ip>\admin$ as <service-account>
System error 5 has occurred. Access is denied.

(Imaging itself doesn't need this — domain join and offline provisioning run on the server, which is why a machine can image fine yet fail PXE Reboot.)

The account you grant local admin must be the exact same account configured in Settings → Active Directory → Service account. If the error shows one name (e.g. a UPN like svc-deploy@corp.example.com) but you granted a differently-named account local admin, the authorised account isn't the one Deploy Manager uses — System error 5 with a correct password means the account authenticated but isn't a local admin. Confirm both refer to the same AD account (same SID), not just similar names.

Grant it fleet-wide with Group Policy Preferences (the additive method — it adds the account without removing existing local admins):

1
Create/edit a GPO In Group Policy Management, link a GPO to the OU holding your target computers.
2
Add a Local Group preference Computer Configuration → Preferences → Control Panel Settings → Local Users and Groups → New → Local Group.
3
Configure it Action: Update. Group name: Administrators (built-in) (this maps to the well-known SID S-1-5-32-544, so it's language-independent). Leave "Delete all member users / groups" unticked — ticking them wipes existing local admins. Under Members → Add, add the service account as DOMAIN\ServiceAccount with action Add to this group.
4
Apply and verify gpupdate /force on a target (or wait for refresh), then confirm with net localgroup Administrators — the service account should be listed. Retry PXE Reboot.
The target must have applied this policy before you use remote reboot on it. These actions operate on a machine that's already running Windows and domain-joined (they reboot it into the deployment), so the policy lands on its normal refresh. Brand-new bare-metal machines are PXE-booted manually instead (BIOS boot menu), so they don't need this.
Local admin on every target is a privileged, fleet-wide grant — if the account is compromised it is admin on all those machines. It's inherent to the admin$/WMI mechanism (there's no lower-privilege way to write to an admin share). Scope the GPO to only the OUs where you use remote reboot, keep the account's password strong (encrypted at rest in Deploy Manager) and its privileges minimal elsewhere. If you'd rather not grant standing admin, PXE-boot targets manually instead of using the remote-reboot buttons.

WinRM for machine lookup (optional)

The Look up button on the Add Machine page queries a running target over WinRM (port 5985) to auto-fill its MAC address. If WinRM isn't enabled on the target you'll see "WinRM cannot complete the operation… WinRM may not be enabled/reachable on the target (5985)" — just enter the MAC manually, or enable WinRM fleet-wide by GPO. This is a convenience only; PXE Reboot / Pre-stage Reboot do not use WinRM (they use admin$/WMI).

Enable WinRM by GPO — three pieces, all configurable centrally:

1
Enable the listener Computer Config → Policies → Administrative Templates → Windows Components → Windows Remote Management (WinRM) → WinRM Service → "Allow remote server management through WinRM" = Enabled, IPv4/IPv6 filter *.
2
Start the service Computer Config → Policies → Windows Settings → Security Settings → System Services → Windows Remote Management (WS-Management) = Automatic.
3
Open the firewall Computer Config → Policies → Windows Settings → Security Settings → Windows Defender Firewall with Advanced Security → Inbound Rules → New Rule → Predefined → Windows Remote Management → allow Windows Remote Management (HTTP-In) (5985).

Then gpupdate /force and reboot a target; test from the server with Test-WSMan <target>.

Re-imaging existing machines — computer account re-use (KB5020276)

When you re-image a machine whose computer object already exists in AD, Deploy Manager re-uses that object (it resets the account's password rather than deleting and recreating it). Since the Microsoft hardening update KB5020276, some domain controllers block this re-use. When that happens, imaging succeeds but the machine can't domain-join, and the server log records:

Offline domain-join provisioning failed for <host>:
NetProvisionComputerAccount returned 2732 (0x00000AAC)

Error 2732 is NERR_AccountReuseBlockedByPolicy. Deploy Manager provisions the account offline (NetProvisionComputerAccount, impersonating the service account) rather than doing an online domain join, and this offline re-use is hardened inconsistently across domain controllers — depending on the DC's patch level neither the "Domain controller: Allow computer account re-use during domain join" allow-list GPO, Full Control, nor object ownership reliably lifts the block. Rather than have you chase a DC-side policy that may or may not apply, Deploy Manager handles it directly.

Recommended fix — enable automatic recreate

In Settings → Active Directory, enable "Recreate computer object when re-use is blocked". When a re-image hits error 2732, Deploy Manager deletes the stale computer object (using the service account's existing delete rights) and recreates it — a fresh create has no re-use gate, so the join proceeds. This works regardless of DC patch level or policy, which is why it is the recommended approach for a mixed fleet.

Deleting a computer object also removes its escrowed BitLocker recovery keys and security-group memberships. The recreate only triggers when a re-use is actually blocked (error 2732), and the object is recreated immediately in the same OU — but if you rely on BitLocker key escrow or group-based targeting, weigh this before enabling it, and re-escrow keys / re-add group membership after imaging. If your DCs do not block re-use, leave it off and the object is preserved as normal.

Alternatives (manual)

If you would rather not enable the setting, you can clear a blocked machine by hand:

  • Delete the stale object and re-image — Deploy Manager then creates a fresh object it owns. The service account already has delete rights from the delegation above.
  • Add the service account to the allow-list GPO — Computer Config → Policies → Windows Settings → Security Settings → Local Policies → Security Options → "Domain controller: Allow computer account re-use during domain join", linked to the Domain Controllers OU. This is Microsoft's sanctioned mechanism, but note it governs the online join path and does not lift the block on Deploy Manager's offline provisioning on every DC build — test it in your environment before relying on it.
Machines Deploy Manager itself creates are provisioned by the service account, so on DCs that honour the creator/owner check they re-image without any of this. The automatic-recreate setting is the portable option for environments where offline re-use is blocked outright.

WinPE local account (Settings)

Set the local administrator account name and password that WinPE bakes into deployed machines via unattend.xml. The password is never served over the network — it is embedded into boot.wim when you patch it (section 11), so re-patch whenever you change it.

Email notifications (Settings)

Configure the SMTP host, port, credentials, from-address and a notify address to receive an email when a deployment completes or fails. Notifications can be toggled per event (on complete / on error), and the Send test email button verifies the configuration before you rely on it. The SMTP password is encrypted at rest.

Software delivery — BranchCache (Settings)

For multi-site WANs, enabling BranchCache changes how newly imaged machines download software packages: they use BITS and activate Windows BranchCache Distributed Cache mode, so machines at the same site serve cached packages to each other instead of re-downloading over the WAN. No hosted-cache server or WSUS infrastructure is required.

At least one BranchCache-enabled machine must already be online at a site before the next machine images there — the first machine at each site always downloads from the server.

Intune Autopilot (Settings)

Enables automatic Autopilot device registration via the Microsoft Graph API after each workgroup deployment. Requires an Entra app registration with an additional Graph permission — covered in full in sections 18–19.

9iPXE binaries & TFTP root

Deploy Manager uses iPXE to move the boot process off slow TFTP and onto HTTP as early as possible.

Nothing to do here — the installer already staged these files. The signed iPXE Secure Boot chain and wimboot ship inside the MSI and are placed in the TFTP root during installation. This section is reference; skip to section 10 (DHCP) unless you want to update the binaries.

To pull a newer signed iPXE release later (optional), run in an elevated PowerShell on the server:

powershell -ExecutionPolicy Bypass -File "C:\Program Files\DeployManager\scripts\Get-IpxeBinaries.ps1"

It downloads the latest signed chain and overwrites the bundled copies in place. The version that shipped is recorded in C:\ProgramData\DeployManager\tftp\VERSION.txt.

The files staged in the TFTP root:

FilePurpose
ipxe-shim.efiUEFI boot file (DHCP option 67). A Microsoft-signed shim; loads ipxe.efi. Works with Secure Boot on or off.
ipxe.efiiPXE with the full driver set, signed by the iPXE Secure Boot CA. Loaded by the shim — not booted directly.
snponly-shim.efi / snponly.efiAlternative chain using iPXE's UEFI SNP-only driver. Try this if a NIC misbehaves with the full ipxe.efi.
undionly.kpxeiPXE for legacy BIOS clients (no Secure Boot involved)
wimbootSecure Boot-capable WIM loader — loads WinPE over HTTP

When an iPXE client asks the server for http://<server>:8080/boot.ipxe, Deploy Manager generates the boot script on the fly from current settings — there is no boot script file to maintain.

Final TFTP root layout (C:\ProgramData\DeployManager\tftp):

tftp\
├── ipxe-shim.efi      # UEFI boot file (Microsoft-signed shim)
├── ipxe.efi           # loaded by the shim (iPXE-signed)
├── snponly-shim.efi   # alt chain
├── snponly.efi
├── undionly.kpxe      # legacy BIOS
├── wimboot
└── Boot\
    ├── BCD
    ├── boot.sdi
    └── boot.wim
UEFI Secure Boot works out of the box — no firmware changes, no key enrollment, no disabling Secure Boot. Since iPXE v2.0.0 the project ships a Microsoft-signed shim (ipxe-shim.efi, signed by the "Microsoft Windows UEFI Driver Publisher" CA that firmware already trusts). The shim loads ipxe.efi, which is signed by the iPXE Secure Boot CA embedded in that shim. The trust chain is:
Firmwaretrusts Microsoft UEFI CA
→
ipxe-shim.efiMicrosoft-signed
→
ipxe.efiiPXE-CA signed
→
wimboot → WinPEMicrosoft-signed boot mgr

Point DHCP option 67 at ipxe-shim.efi for UEFI clients and you are done. The one trade-off: you must use the iPXE binaries exactly as shipped — you cannot compile a custom iPXE with extra features and keep it Secure Boot-trusted, because the shim only trusts binaries the iPXE project itself signs. The stock ipxe.efi covers everything Deploy Manager needs.

10DHCP configuration

Only two DHCP scope options are required. The installer pre-stages an autoexec.ipxe script in the TFTP root that automatically chains iPXE to the Deploy Manager HTTP boot script, so no DHCP user-class policies or vendor-class rules are needed.

1
Set the scope options On the DHCP scope that covers the machines you want to PXE boot, in an elevated PowerShell on the DHCP server: 
# Replace 192.168.10.0 with YOUR scope and <server-ip> with the Deploy Manager server IP
Set-DhcpServerv4OptionValue -ScopeId 192.168.10.0 -OptionId 66 -Value "<server-ip>"
Set-DhcpServerv4OptionValue -ScopeId 192.168.10.0 -OptionId 67 -Value "ipxe-shim.efi"
OptionValuePurpose
066 — Boot Server Host NameThe Deploy Manager server's IP addressTells PXE where to TFTP from
067 — Bootfile Nameipxe-shim.efiMicrosoft-signed shim for Secure Boot
Use ipxe-shim.efi, not ipxe.efi, as the UEFI boot file. The shim is the Microsoft-signed first stage that Secure Boot firmware will accept; it then loads ipxe.efi itself. Pointing option 67 straight at ipxe.efi would be rejected on machines with Secure Boot enabled.
2
Verify Confirm the options are set: 
Get-DhcpServerv4OptionValue -ScopeId 192.168.10.0
You should see option 66 = your server IP and option 67 = ipxe-shim.efi.
3
Mixed UEFI/BIOS fleets (optional) Use DHCP policies on option 60/93 (client architecture) to serve ipxe-shim.efi to UEFI and undionly.kpxe to BIOS within the same scope.

How the boot chain works

No manual intervention needed — this is just background on what happens during PXE boot:

  1. Firmware PXE sends DHCP, gets option 66 (server IP) and option 67 (ipxe-shim.efi).
  2. Firmware downloads ipxe-shim.efi via TFTP — Microsoft-signed, passes Secure Boot.
  3. The shim loads ipxe.efi — signed by iPXE CA (trusted by the shim).
  4. iPXE fetches autoexec.ipxe from the TFTP server (pre-installed by the MSI).
  5. autoexec.ipxe chains to http://<server>:8080/boot.ipxe (using DHCP option 66 as the server address).
  6. The HTTP boot script downloads wimboot, BCD, boot.sdi, and boot.wim over HTTP at full LAN speed.
  7. WinPE loads and runs Start-OSDeploy.ps1.
Test: PXE-boot any machine on the scope. You should see iPXE load, fetch boot.ipxe, download boot.wim over HTTP at LAN speed, and land in WinPE. Within a minute the machine appears under Machines in the web UI.

11Patch boot.wim

Start-OSDeploy.ps1 inside the boot image ships with two placeholders: the server URL and the WinPE local password. They must be patched before the image is usable — otherwise WinPE has no idea where its server is.

1
Check prerequisites In the web UI open Settings → Boot Image. The status strip must show boot.wim found, API URL configured, WinPE password configured and ADK DISM found.
2
Run the patch Click Patch boot.wim. The server mounts the image with ADK DISM (mount point C:\WinPE_MOUNT), replaces the placeholders, and commits. Output appears on the page.
The patch also copies the latest Start-OSDeploy.ps1 and Start-PostInstall.ps1 into the WIM before replacing placeholders — so after an upgrade you only need to re-patch (no manual file copy).
Re-run the patch whenever the API server URL or the WinPE local password changes, or after upgrading Deploy Manager. The equivalent command line, if you prefer scripting: 
powershell -ExecutionPolicy Bypass -File "C:\Program Files\DeployManager\scripts\Update-BootWim.ps1" `
    -ServerUrl "https://<server-ip>:8090" -WinpePassword "<password>"

1

14Software & packages

Software entries describe a single installer; Packages group them into an ordered set applied after Windows setup.

1
Add software items Under Software → Add Software, for each application provide the installer location (a UNC path or a path under the images share), the working directory, and the silent install command, e.g. setup.exe /S or msiexec /i app.msi /qn.
2
Group them into a package Under Packages → Create Package, select the software items and their order, e.g. a "Standard SOE" package with your antivirus, office suite, VPN client and monitoring agent.

During deployment, each install's exit code is captured and reported to the job — failures are visible per-application in the Jobs page.

Each installer runs under a timeout (default 60 minutes, set under Settings → Software Delivery). If an installer hangs past the limit it is terminated, recorded as a timeout failure on the Jobs page, and the deployment continues to the next package — one bad installer can't stall the whole machine. Raise the limit for large/slow installers; the change takes effect after the next boot.wim patch. Tip: for PowerShell-script installers, include -NonInteractive in the install command so the script errors out instead of blocking on a prompt.

BitLocker drive encryption (Settings)

Deploy Manager can turn on BitLocker automatically at the end of the post-install phase. Enable it under Settings → BitLocker. Requires a TPM (present on all Secure Boot hardware); a machine without a ready TPM is skipped and reported, never failed.

OptionWhat it does
Volumes to encryptOS drive only; OS + all fixed data drives (auto-detects every fixed, non-OS volume — no drive letters needed); or Specific volumes (semicolon-separated labels or letters). Removable/USB drives are never encrypted.
Encryption methodXTS-AES 256 (recommended) or 128.
Used space onlyEncrypts only used space — fast, ideal for a fresh image (new data is encrypted as written). Turn off for full-disk.
Recovery-key escrowBack up each recovery key to any combination of: Active Directory (domain-joined), Entra ID / Intune (Entra/hybrid-joined machines — a pure Autopilot device isn't Entra-joined until OOBE, so Intune policy usually owns BitLocker for those, and Deploy Manager skips the Entra backup on a device that isn't Entra-joined), and/or a network share (writes <ComputerName>_<drive>.txt; the machine account needs write access). The share stores the key in clear text — lock it down to admins.
AD backup is handled by my Group PolicyTick this if you already run the GPO “Store BitLocker recovery information in Active Directory Domain Services.” That policy escrows the key at encryption time and refuses the manual backup cmdlet — so without this option Deploy Manager would report a false failed. The OS drive is encrypted first with a TPM protector (transparent unlock at boot) plus a recovery password; each fixed data drive then gets a recovery password and auto-unlock (it unlocks once the OS drive has). Per-machine status shows on the Jobs page as a BitLocker badge (protected / failed / skipped), with per-volume detail on hover.

Recovery keys are never written to the deployment logs — only the key protector ID is. Look up a machine's recovery key by its computer name in AD (or your Intune portal), like any other BitLocker device. Enabling BitLocker takes effect after the next boot.wim patch.
Your recovery keys are only as safe as your escrow. Confirm keys are actually landing in AD (check a test machine's computer object for the BitLocker Recovery tab) before relying on it fleet-wide — a drive encrypted with a key you can't recover is a permanent lockout.

15Deploying a machine

Every deployment follows the same basic steps, regardless of join mode. The sections after this one walk through the two deployment scenarios in detail: domain join and Autopilot / Intune.

1
Register the machine PXE-boot the target machine once (or add it manually under Machines → Add Machine with its MAC address). Discovered machines report their MAC, IP, model and serial number automatically.
2
Create the job Go to Deploy → New Deployment: pick the machine, confirm the device name (prefix is suggested), site and OU (auto-detected from subnet), choose the WIM image, driver package, software package (optional), and the join mode:
  • Domain Join — pre-stages the computer object in AD using offline domain join. The machine boots to the Windows login screen, already on the domain.
  • Workgroup (Autopilot) — joins a workgroup instead of the domain. The machine boots to the Out-of-Box Experience (OOBE) for Autopilot / Entra enrollment. You can set a Group Tag to control which Autopilot deployment profile applies.
Then choose an action:
  • Save Job Only — saves the job file. You manually PXE-boot the machine (e.g. via BIOS boot menu).
  • Save Job + PXE Reboot — saves the job, then remotely sets the machine's next boot to its PXE/NIC firmware entry and reboots it via WMI. Requires the service account to have local admin rights on the target.
  • Save Job + Pre-stage Reboot — saves the job, then remotely stages WinPE to the machine's local disk and reboots into it via the Windows Boot Manager. No PXE required — works over the WAN and on models where PXE is unavailable or blocked by Secure Boot policy. Also requires local admin on the target.
3
WinPE loads and images the machine It finds its pending job within seconds and starts imaging — partitioning, applying the WIM, injecting drivers, joining the domain or workgroup, writing unattend.xml with the site's timezone/locale, then rebooting into Windows Setup.
4
Post-install phase On first boot, a SetupComplete.cmd script runs Start-PostInstall.ps1, which:
  • Installs any software from the assigned package (with per-app progress reported to the Jobs page)
  • Installs the driver package via pnputil (if one was assigned)
  • For Autopilot jobs: collects the hardware hash, serial number and Windows Product ID, submits them to the server. If automatic Intune registration is enabled, the server registers the device with Intune immediately.
  • Marks the job as complete (or error, if any software failed)
5
Watch it live The Jobs page streams status in real time: pending → imaging → software installs (per-app progress) → complete. A watchdog times a job out if it goes too long with no activity (progress or heartbeat) — see Settings → Job Monitoring — so a hung machine can't stay "imaging" forever, while a slow-but-progressing one is left alone. Email notifications on completion/failure can be enabled in Settings.

16Domain join deployment

Use this mode when the machine will be managed by on-premises Active Directory and Group Policy.

Prerequisites

  • AD service account configured in Settings with Full Control on Computer objects in the target OUs (section 6)
  • At least one site configured under Settings → Sites with the correct OU
  • A Windows WIM image registered under Images

Walkthrough

1
Create the deployment job Go to Deploy → New Deployment. Select the machine, enter the device name (e.g. CORP-PC042), and verify the site/OU are auto-detected correctly. Set join mode to Domain Join.
2
Assign content Choose the WIM image, driver package (if you have one for the model), and a software package (e.g. your standard SOE build with AV, Office, VPN).
3
Start imaging Click Save Job + PXE Reboot (if the machine is online) or Save Job Only and PXE-boot manually.
4
Machine images and reboots After Windows Setup completes, the post-install phase installs software and reports results. The machine arrives at the Windows login screen, already domain-joined in the correct OU with the site's timezone and locale applied.
The job turns green on the Jobs page. The machine is on the domain, Group Policy will apply on next logon, and software is installed.

Re-imaging an existing domain machine

Deploy Manager detects if the computer object already exists in AD. If it does, the existing OU is preserved (the machine is re-joined in place) rather than being moved to the site's default OU. This prevents machines from unexpectedly changing OUs during a re-image.

Some domain controllers block re-use of an existing computer object since KB5020276. If a re-image fails to join the domain with error 2732, enable "Recreate computer object when re-use is blocked" in Settings — see computer account re-use (section 6).

17Autopilot / Intune deployment

Use this mode when the machine will be managed by Microsoft Intune via Autopilot. The machine boots to the Out-of-Box Experience (OOBE) where the user signs in with their Entra ID credentials to complete enrollment.

Prerequisites

  • An Intune tenant with an Autopilot deployment profile configured (user-driven or self-deploying, depending on your scenario)
  • A Windows WIM image (a clean, generic Windows 11 Pro/Enterprise image works best)
  • Internet connectivity from the target machine after imaging (for Autopilot enrollment at OOBE)

Walkthrough

1
Create the deployment job Go to Deploy → New Deployment. Select the machine, enter the device name, and set join mode to Workgroup (Autopilot).
2
Set a group tag (optional) The group tag controls which Autopilot deployment profile is assigned to the device. Leave blank if your Intune tenant has a single default profile, or enter the tag that matches your profile assignment rules (e.g. Standard, Kiosk, SharedDevice).
3
Assign content Choose the WIM image and driver package. A software package is optional — for Intune-managed machines you will typically deploy software via Intune rather than the Deploy Manager package. Leaving the package empty is fine; the system skips software installation cleanly.
4
Start imaging Click Save Job + PXE Reboot or PXE-boot manually.
5
Post-install collects hardware details After Windows Setup completes, the post-install script collects:
  • The hardware hash (from MDM_DevDetail_Ext01)
  • The serial number (from Win32_BIOS)
  • The Windows Product ID (from the registry)

These are submitted to the Deploy Manager API. On the Jobs page, the Autopilot badge turns green once the hash is collected.

6
Register the device in Intune There are two options:
  • Automatic (recommended) — enable automatic Intune registration in Settings. Deploy Manager registers the device via the Graph API immediately after collecting the hash. The Jobs page shows the Intune status in real time (importing → registered → profile assigned).
  • Manual — download the Autopilot CSV from the Jobs page (per-device or bulk export) and import it into the Intune portal under Devices → Enroll devices → Windows Autopilot devices → Import.
7
Machine boots to OOBE After the post-install phase completes, the machine reboots into the Windows Out-of-Box Experience. If the device is registered in Intune with a profile assigned, OOBE shows the Autopilot branding (your organisation's logo and name). The user signs in with their Entra ID credentials to complete enrollment.
The device must be registered in Intune with a deployment profile assigned before the user reaches OOBE. If using manual CSV import, allow up to 40 minutes for Intune to process the import and assign a profile. With automatic registration enabled, Deploy Manager polls for profile assignment and shows the status on the Jobs page — wait for the Intune profile assigned badge before handing the machine to the user.

Autopilot CSV format

The CSV exported by Deploy Manager follows the standard Intune import format:

Device Serial Number,Windows Product ID,Hardware Hash,Group Tag
ABC1234567,12345-OEM-1234567-12345,AQAA9f+L...base64...,Standard

You can download a CSV for a single device (click the green CSV badge next to the Autopilot indicator on the Jobs page) or for all devices with collected hashes (click the Autopilot CSV button at the top of the Jobs page).

18Entra app permissions for Intune

To use automatic Intune registration, your Entra ID app registration needs an additional API permission. If you are only using Entra SSO (sign-in to the web UI), no changes are needed — this section is only required for the automatic Autopilot registration feature.

1
Open your app registration In the Entra admin centre, go to App registrations and open the app you created for Deploy Manager.
2
Add the API permission Go to API permissions → Add a permission → Microsoft Graph → Application permissions. Search for and add: 
DeviceManagementServiceConfig.ReadWrite.All
This is an application permission (not delegated). It allows the server to manage Autopilot device identities without a signed-in user.
3
Grant admin consent Click Grant admin consent for <your tenant> and confirm. The permission status should change to Granted for <tenant>.
4
Verify existing credentials Your app registration should already have a Tenant ID, Client ID, and Client Secret configured in Deploy Manager under Settings → Entra ID (these are the same credentials used for SSO). If you have not set up Entra SSO yet, enter the Tenant ID, Client ID and create a client secret in Entra, then paste it into Settings. The secret is encrypted at rest.
The Entra app now has permission to read, create and update Autopilot device identities via the Microsoft Graph API. Proceed to enable the feature in Deploy Manager Settings.

19Automatic Autopilot registration

When enabled, Deploy Manager automatically registers devices with Intune Autopilot as soon as the hardware hash is collected during post-install. This eliminates the manual step of downloading a CSV and importing it into the Intune portal.

Enable the feature

1
Complete the Entra app setup Follow section 18 — the DeviceManagementServiceConfig.ReadWrite.All permission must be granted with admin consent.
2
Enable the toggle Go to Settings → Intune Autopilot, tick Automatically register devices in Intune Autopilot, and save.
3
Re-patch boot.wim Go to Settings → Boot Image and click Patch boot.wim. This bakes the latest post-install script (which now sends the serial number to the API) into the boot image.
The next Autopilot deployment will automatically register the device in Intune.

How it works

When the post-install script submits the hardware hash, serial number and product ID to the Deploy Manager API, the server checks if automatic registration is enabled. If it is:

  1. Checks Intune — queries the Graph API by serial number to see if the device is already registered as an Autopilot device.
  2. If already registered — skips import. If the group tag in Deploy Manager differs from the one in Intune, the server updates the group tag in Intune to match.
  3. If not registered — imports the device into Intune with the hardware hash, serial number and group tag. Polls for import completion (typically 1–15 minutes).
  4. Polls for profile assignment — after the device is registered, continues polling Intune until a deployment profile is assigned (typically 1–10 minutes after import).

Intune status on the Jobs page

The Jobs page shows the Intune registration status in real time alongside the Autopilot badge:

StatusMeaning
Checking Intune...Querying Graph API to see if the device already exists
Importing to Intune...Import request submitted; waiting for Intune to process it
Intune registeredDevice is registered in Autopilot but no deployment profile assigned yet
Intune profile assignedA deployment profile is assigned — the machine is ready for Autopilot enrollment at OOBE
Intune errorRegistration failed — check the Deploy Manager service logs in Event Viewer for details
The registration process runs in the background on the server. The machine continues its post-install phase and may reboot to OOBE before registration is complete. If you are using a user-driven Autopilot profile, the user will be prompted to sign in at OOBE — they should wait until the Jobs page shows Intune profile assigned before signing in, otherwise OOBE will not show the Autopilot experience. If using a self-deploying profile, the machine will retry automatically until the profile is available.

20HTTPS certificate

A 5-year self-signed certificate is generated on first start. To replace it with one signed by your internal CA (recommended — removes browser warnings and enables full certificate validation from WinPE):

1
Generate a CSR Under Settings → Certificate → Generate CSR, enter the server's DNS name(s) and IP(s) as subject alternative names. The private key never leaves the server (stored encrypted until the signed certificate is installed).
2
Sign the CSR Sign with your CA — for AD CS: 
certreq -submit -attrib "CertificateTemplate:WebServer" deploymanager.csr
3
Upload the signed certificate Upload the certificate (PEM/Base64) plus any chain on the same page. It is served to new connections immediately — no service restart needed. The dashboard warns 30 days before expiry.

21Users & authentication

  • Breakglass login — the password set during first-run setup, always available at /auth/login. Login attempts are rate-limited (10 per 15 minutes per IP).
  • Microsoft Entra ID SSO — configure Tenant ID, Client ID and Client Secret in Settings (create an app registration with redirect URI https://<server>:8090/signin-oidc). If you access the server via multiple hostnames or IPs, add a redirect URI for each (e.g. https://deploymanager.example.com:8090/signin-oidc and https://192.168.1.10:8090/signin-oidc). Only users you have added under Users (by UPN) can sign in, each with a role controlling their access.
    Restart the Deploy Manager service after changing the sign-in method, Tenant ID, Client ID or secret. These settings are read once at startup and do not take effect until the service is restarted (via the tray app or sc.exe restart DeployManager).
  • Audit log — every sign-in, settings change, certificate operation and deployment action is recorded, viewable under Audit.

The stuck-job watchdog now times out on inactivity (no progress or heartbeat) instead of total elapsed time — so a legitimately long deployment is never falsely failed, and a real hang is caught within minutes. WinPE and PostInstall send heartbeats during long steps (WIM download/apply, long installers). Thresholds are configurable under Settings → Job Monitoring. 1.19.0BitLocker drive encryption after imaging. 1.18.3Reliable software-install exit codes + bounded drive mapping.
